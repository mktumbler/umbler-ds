A ordem das seções **não é decorativa** — trocar a ordem (ou pular um estágio) quebra a lógica de persuasão e custa conversão.

## Sequência por estágio de awareness

### Unaware → Problem-aware

Tráfego frio, descobrindo o problema (ex: clicou em ad de "WhatsApp Business" e não conhece a Umbler).

```
Hero → Logo Cloud → Dores/Problemas → Solução → Features →
Social Proof → Pricing → FAQ
```

### Solution-aware

Já sabe que precisa, está comparando opções (ex: tráfego de "alternativa CRM").

```
Hero → Logo Cloud → Diferenciação → Features detalhadas →
Integrações → Pricing → FAQ
```

### Product-aware

Já conhece a Umbler (ex: remarketing, email, indicação).

```
Hero → Pricing → Features → FAQ → CTA
```

<Metodologia title="Por que a sequência muda com o awareness">

A LP serve a tráfego de **estados diferentes** — quem nunca ouviu falar da Umbler vs. quem já comparou. A sequência muda conforme o estado.

No tráfego **unaware/problem-aware**, o **agitate problem** (dor explícita) é central. Sem ele, o leitor não tem motivo para se importar com a solução.

No **solution-aware**, a diferenciação vem antes das features — o leitor precisa entender **por que a Umbler especificamente**, antes de avaliar o que ela faz.

No **product-aware**, pricing entra logo após o hero porque a dúvida residual é "vale a pena pra mim?", não "o que é?".

</Metodologia>

## Lógica de persuasão (páginas importantes)

Em páginas que precisam **levar à conversão alta** (LP de produto, LP de pricing), a sequência canônica:

1. **Promessa clara** (hero)
2. **Explicação rápida** (o que é)
3. **Prova / social proof** (não estou sozinho)
4. **Dor / problema** (eu tenho isso)
5. **Solução e mecanismo** (como resolve)
6. **Benefícios** (no meu mundo)
7. **Casos de uso** (assim funciona pra mim)
8. **Prova adicional** (mais um sinal)
9. **Redução de objeções** (FAQ)
10. **CTA**

<Metodologia title="Por que pular um passo custa conversão">

Cada passo prepara o próximo. **Pular** um custa — se o leitor não viu prova suficiente antes do mecanismo, a solução soa fraca. Se viu dor sem prova depois, fica cético.

</Metodologia>

## Tension-Release Rhythm

A página alterna tensão (dor, medo, status quo) e alívio (solução, prova, outcomes):

```
Hero                      → release (promessa)
Social Proof              → release (credibilidade)
Problem Agitation         → TENSÃO (dor do status quo)
Solution / Features       → release (alívio)
Manifesto                 → TENSÃO emocional (ambição)
Product Showcase          → release (prova visual)
Pricing                   → neutro (decisão)
FAQ                       → release (objeções resolvidas)
CTA Banner                → TENSÃO (loss aversion + urgência)
```

<Metodologia title="Por que a alternância tensão/alívio mantém leitura">

Esse ritmo mantém engajamento. LP de uma nota só (toda alívio ou toda tensão) cansa. Alternância gera ritmo de leitura.

</Metodologia>

## Padrões de seção comprovados

### Padrão "Problema → Solução"

Lado esquerdo: dor real do cliente (em aspas, voz dele). Lado direito: "Com Umbler" — como resolve. Repetido 4-5 vezes na seção.

### Padrão "Steps 01 / 02 / 03"

Configuração ou ativação em 3 passos numerados. Visual: número grande + título + descrição curta.

### Padrão "Cards de Segmento"

Grid de cards por indústria + descrição rápida de como funciona ali. Cada card tem ícone, título do setor, descrição.

### Padrão "Comparação lado a lado"

Tabela ou cards comparando Umbler vs. alternativa. Cada lado com bullets de características. Tom: "não é melhor nem pior, é diferente".

<Metodologia title="Por que esses 4 padrões convertem (e onde foram testados)">

**Problema → Solução.** Visitante se identifica antes de ouvir a solução. Cada par é um micro-argumento completo. Foi testado em LP de CRM.

**Steps 01/02/03.** Reduz a objeção "deve ser complicado". 3 é o número mágico — 4 já parece muito, 2 parece simples demais pra ser verdade. Testado em LP de Agente IA.

**Cards de Segmento.** Visitante encontra "seu" setor e se sente atendido. Testado em LP de Chatbot.

**Comparação lado a lado.** Educa e posiciona. O leitor sai sabendo escolher (e a Umbler aparece como a única que tem **as duas coisas**). Testado em LP de Agente IA (vs. Chatbots).

</Metodologia>

## Mobile-first

A LP é lida majoritariamente no celular. Adaptações:

- **CTA sempre visível** (sticky navbar ou floating button no fim do scroll)
- **Hero:** texto acima, visual abaixo
- **Seções longas** colapsam (accordion) ou simplificam (menos features visíveis)
- **Pricing:** cards empilhados, não lado a lado
- **Logo cloud:** scroll horizontal funciona bem
- **Tipografia:** não encolher tanto a ponto de perder a hierarquia — display continua grande

### Anti-pattern mobile

- Hero com 3 CTAs lado a lado
- Tabela de comparação que vira scroll horizontal feio
- Manifesto com texto enorme que ocupa 3 telas inteiras
- Logo cloud em grid 5×5 que reduz cada logo a 20px

Se mobile parece pior que desktop, a estrutura foi otimizada pra apresentação, não pra leitura. Refaça.
